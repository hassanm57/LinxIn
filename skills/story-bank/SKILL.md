# story-bank

Your career's proof, written once and reused everywhere. `cover-letter`, `proposal`, and
`interview-prep` all pull from here, so the more you bank, the less you rewrite.

## What it manages

`profile/stories.md` — a growing set of narratives in **STAR + Reflection** form
(Situation, Task, Action, Result, Reflection), each with tags for retrieval. Follows
`profile/stories.template.md`.

## Modes

### Add / capture
1. Take a raw experience the user describes (or one surfaced during `cover-letter` /
   `interview-prep`).
2. Draw out anything missing by asking — especially the **Result** (a metric if one exists) and
   the **Reflection** (what they learned / would do differently). The Reflection is what makes a
   story land; don't skip it.
3. Write it to `profile/stories.md` with a short title and consistent tags
   (e.g. `leadership`, `backend`, `incident`, `ambiguity`, `conflict`).

### Retrieve
Given a role, a competency, or a tag, return the best-matching stories for another skill to use.
Prefer stories with concrete results and clear ownership ("I", not "we").

### Curate
De-duplicate near-identical stories, tighten bloated ones, and keep tags consistent so retrieval
stays reliable.

## Guardrails

- **Never invent stories, metrics, or outcomes.** Capture only what the user actually did. If a
  detail is missing, ask or leave it marked — don't fill it in.
- Keep the user's voice; STAR is a scaffold, not a script that flattens how they sound.
- `profile/stories.md` is personal data — it stays local (gitignored).
